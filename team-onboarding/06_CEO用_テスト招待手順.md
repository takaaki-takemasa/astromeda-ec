# CEO 用 テスト招待手順 (1人分・所要 30 分)

このドキュメントは **CEO が実際にメンバー 1 人を招待して onboarding を全部通し、詰まりを発見する** ためのチェックリストです。

---

## なぜこれが必要か

ここまで A 演習 (疑似メンバー) と B 監査 (構造調査) を実施し、文書を patch 0155 で改訂しました。
ただし **「実機で別 PC・別アカウントから本当に通せるか」** の最終確認はまだです。

CEO 1 人で 30 分使って 1 度通せば、残る穴 (UI 細部・CEO 判断の必要箇所) が全て見えます。

---

## 招待相手の選び方

理想順位:
1. **家族・親族で IT 苦手な人** (PC は普通に使えるが git/プログラミングは知らない、を想定する Astromeda 想定メンバー像と一致)
2. **Astromeda の実マーケ担当 1 名** (将来本当に使う人)
3. **CEO 自身が普段使わない別 PC で別 GitHub アカウント** (最低限のテスト・代替案として)

---

## 30 分のスクリプト

### 0 分: 招待相手に伝える

LINE/メール で:
```
これからテストで Astromeda 運用システムへの参加手順を試したいです。
30 分時間ください。

最初に、GitHub アカウントを作っていない場合は作ってもらいます。
案内に従って進めるだけ・難しい操作はないです。

PC で時間取れる時に「準備できた」と返信してください。
```

### 5 分: 相手から GitHub username を受け取る

```
GitHub username を教えてください (例: tanaka-marketing)
```

### 10 分: CEO 側 — GitHub Collaborator として招待

ブラウザで:
```
https://github.com/takaaki-takemasa/astromeda-ec/settings/access
```

1. 「Add people」緑ボタン
2. username 入力
3. 権限: **Write** を選択
4. 「Add to this repository」

→ 招待メールが相手に届く。

### 12 分: 相手にセットアップ手順書を送る

LINE/メール添付で:
- `01_メンバー向け_セットアップ手順.md` (v4)
- `02_絶対にやらないこと.md`
- `04_admin_password_運用ルール.md`
- `05_コード変更vs管理画面で完結する変更.md`
- `新IPコラボ出品_完全手順書.html`

メッセージ:
```
GitHub の招待メールが届いたら、Accept を押してください。
そのあと、添付の「01_メンバー向け_セットアップ手順.md」を最初から進めてみてください。
30 分で終わるはずです。

Cowork (Claude Desktop) は claude.ai Pro ($20/月) が必要なので、
登録は私が立て替えます (経費精算します)。

詰まったら、いつでも LINE/電話で連絡してください。
全部の手順を辿ってもらって、どこで止まったか教えてもらうのが目的です。
```

### 12〜25 分: 相手が onboarding 実施 (CEO は待つ)

CEO は別の作業をしながら、LINE で「進捗どうですか?」と 5 分おきに聞く。
詰まったら即サポート (画面共有 or 電話)。

### 25 分: 練習 push が成功したら祝う + フィードバック

相手に:
```
お疲れさまでした！どこで詰まりましたか? 何でも教えてください。
- 画面の文言で分かりにくかったところ
- 手順がない・書いてないけど自分で気付いたこと
- ストレスを感じた箇所

これを CEO 側で改善するのに使います。
```

### 30 分: フィードバックを CEO がメモ → patch 0155-fu に反映

CEO は受け取ったフィードバックを以下のフォーマットで:

```
## (相手の名前) 1 回目 onboarding フィードバック (yyyy-mm-dd)

### 詰まった箇所
- Step X-Y: 「(具体的に何が分からなかったか)」
- 修正案: (何をどう改訂すべきか)

### 良かった点
- (期待通り動いた箇所・分かりやすかった箇所)

### 改善希望
- (相手が「こうだったらいいのに」と思った点)
```

→ Cowork に渡して patch 0155-fu (or 0156) として文書改訂を依頼。

---

## 招待後のクリーンアップ (テスト完了時)

テスト目的で招待した相手で、本格運用しない場合:

1. GitHub Collaborators ページから Remove
2. 相手の PAT も無効化推奨 (相手が `https://github.com/settings/tokens` で削除)
3. Cowork のフォルダ・キャッシュは相手の PC に残しておく (将来また呼ぶかもしれないので)

---

## このテストで見るべき 5 ポイント

CEO が招待中に観察するチェックリスト:

| # | チェック項目 | 合格基準 |
|---|---|---|
| 1 | 相手が PAT 発行画面に詰まらず辿り着けたか | LINE で質問が来ない |
| 2 | Cowork が clone を自動で実行できたか | 相手から「ダウンロードできた」報告がある |
| 3 | astromeda-team skill が clone 後に発動したか | 練習作業の案内が「IP コラボの XXX」のように Astromeda 用語で説明される |
| 4 | 練習 push が 4-5 分後に GitHub commits に反映されたか | 相手が自分の commit を見られる |
| 5 | 相手が「次は admin から商品を編集してみたい」と感想を言うか | 文書の「コード vs admin 使い分け」が伝わっている証拠 |

5/5 なら **本番運用 GO**。
3/5 以下なら patch 0155-fu で更に改訂が必要。
